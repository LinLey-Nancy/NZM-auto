# NZM-auto 项目状态与路线图

更新时间：2026-07-24

## 项目目标

NZM-auto 是一个基于 MaaFramework 的 Windows 桌面自动化工具。最终目标是让用户通过桌面 GUI 创建和运行自定义工作流，包括：

- 模板识别与结果复用
- 鼠标移动、单击、双击和不同鼠标按键
- 单键、组合键与文本输入
- 等待、重试、结果验证和安全停止
- 工作流保存、加载、调试与发布

项目以“输入发送成功不等于目标应用已经处理输入”为安全原则。正式流程应通过模板、截图变化或其他可观测状态验证操作结果。

## 当前架构

```text
src/nzm_auto/
├── application/   CLI 与 GUI 共用的应用服务
├── automation/    第一版模板动作与顺序工作流
├── config/        主配置加载与校验
├── diagnostics/   截图、模板匹配、输入与调试报告
├── gui/           PySide6 桌面工作流编辑器
├── runtime/       MaaFramework 控制器、资源和任务生命周期
├── windowing/     Windows 窗口发现与确定性选择
├── workflow/      工作流 v2 模型、执行器、事件与取消机制
└── cli.py         命令行入口
```

主要边界：

- `application/` 不依赖终端交互，供 CLI 和 GUI 共同调用。
- `workflow/` 负责数据模型、动作执行、结构化事件和协作式取消。
- `gui/` 只负责编辑、展示、确认和线程调度，不直接实现 Maa 动作。
- `runtime/` 统一管理 Maa 控制器连接、分辨率校验、资源加载和释放。

## 已完成阶段

### M1：基础环境与服务层

- 使用 Python 3.12 和 MaaFramework v5.12.1 建立稳定基线。
- 将窗口查询与选择抽离为终端无关服务。
- 新增 `AutomationSession`，统一控制器连接、输入方式覆盖、资源加载和安全释放。
- CLI 已改用应用服务层，原有命令和退出码保持兼容。

### M2：工作流 v2 与键鼠执行器

- 新增严格的工作流 v2 JSON 模型和未知字段拒绝机制。
- 已支持以下步骤：
  - `template_match`
  - `mouse_move`
  - `mouse_click`
  - `key_press`
  - `text_input`
  - `wait`
- 模板识别结果可以写入变量，后续鼠标步骤可以点击匹配框中心。
- 支持步骤启用状态、失败策略、识别重试和间隔。
- 新增结构化工作流事件，便于 CLI 日志和 Qt Signal 展示。
- 新增协作式取消，停止后不会开始下一步输入。
- 新增 `workflow-run-v2` 命令。

### M3：PySide6 桌面 GUI

- 新增 `nzm-auto-gui` 启动入口和可选依赖 `.[gui]`。
- 已实现：
  - 新建、打开、保存和另存工作流
  - 动作组件列表
  - 步骤添加、删除、上移和下移
  - 动态步骤属性编辑
  - 只读目标窗口选择
  - 输入兼容策略选择
  - 运行前目标及风险确认
  - 后台线程执行
  - 实时结构化日志
  - 协作式安全停止
  - 未保存更改确认
- GUI 使用深色开发工具设计系统，支持中文字体、清晰焦点、键盘导航、禁用状态和错误恢复提示。

## 输入兼容性实测

测试目标为标题为“逆战：未来”的文件资源管理器窗口，以及其中打开的 Notepad 测试文档。

| 输入路径 | 实测结果 | 使用建议 |
|---|---|---|
| `Seize` 鼠标双击 | 成功，目标窗口产生明确截图变化 | 游戏前台鼠标默认方案 |
| `PostMessage` 键盘 | Maa Job 成功，但 Notepad 未处理输入 | 不得仅凭 Job 成功判定；游戏模式不默认使用 |
| `Seize + post_input_text` | 字符失真且保存时序不可靠 | 仅保留为可选兼容模式 |
| `Seize + click_key` 离散按键 | 单键和 `Ctrl+S` 等组合键成功 | 适合游戏按键操作 |
| `Seize + key_down/key_up` 逐键文本 | 完整 33 字符与磁盘内容逐字符一致 | ASCII 文本的当前可靠方案 |

当前输入策略：

- `foreground-compatible`：鼠标和键盘使用 `Seize`，游戏默认推荐。
- `background-message`：使用 `PostMessage`，只适合目标应用实测确认支持的场景。
- `driver-interception`：预留 Interception 驱动级输入，需要用户显式安装驱动和管理员权限。

逐键文本目前按 US 键盘虚拟键映射支持 ASCII。中文等 Unicode 文本需要后续增加剪贴板粘贴或输入法专用策略，并进行目标应用验证。

## 当前限制

- GUI 尚不能从截图框选并自动保存模板。
- 目标窗口分辨率仍使用主配置中的严格预期值；普通测试窗口可能需要临时配置。
- 尚无暂停/恢复；当前只有运行、协作式停止。
- 停止只能在安全边界生效，不能中断 MaaFramework 内部正在等待的单个 Job。
- 尚无条件分支、循环、子工作流和跨步骤通用表达式。
- 工作流报告尚未统一覆盖 v2 的全部事件和输入验证结果。
- 模板识别后可点击匹配中心，但 GUI 尚未提供模板预览和单步识别调试。
- GUI 当前以步骤列表为主，不是自由连线流程图。

## 后续阶段计划

### M4：截图、模板与单步调试

目标：让非开发用户可以在 GUI 内完成模板制作和步骤验证。

- 在 GUI 中连接选定窗口并获取截图。
- 自动探测原始分辨率、截图分辨率和 DPI 缩放。
- 提供可缩放截图预览及矩形框选。
- 将裁剪结果保存到当前工作流独立的 `templates/` 目录。
- 在属性编辑器中预览模板、阈值和匹配框。
- 支持“测试当前步骤”和“从当前步骤运行”。
- 展示匹配分数、匹配区域、操作前后截图和差异图。
- 失败时给出可执行的恢复建议，而不是只显示异常文本。

验收标准：

- 用户无需手工编辑 JSON 即可制作模板并完成一次识别。
- 测试步骤不会误执行后续步骤。
- 分辨率不匹配在发送输入前给出明确校准流程。

### M5：工作流控制能力

目标：从线性动作列表升级为可表达常见自动化逻辑的执行器。

- 暂停与恢复。
- 等待模板出现、等待模板消失。
- 条件分支。
- 有上限的循环与超时保护。
- 用户变量和步骤输出引用。
- 子工作流。
- 统一重试、退避和失败处理策略。
- 静态检查不可达步骤、缺失变量和危险无限循环。

验收标准：

- 执行器可表达“识别状态 → 选择分支 → 重试 → 结束”的常见游戏流程。
- 任意循环都有次数或时间上限。
- 停止请求不会导致重复输入。

### M6：验证、诊断与游戏兼容性

目标：让每次输入都能选择适合的验证方式，并形成可审计报告。

- 为输入步骤增加后置验证规则。
- 支持画面变化、模板出现/消失和区域变化验证。
- 增加输入兼容性探针，区分“已派发”和“目标已处理”。
- 检测目标窗口焦点、关闭、最小化、权限等级和分辨率变化。
- 增加 Unicode 文本策略并恢复用户剪贴板。
- 评估管理员权限和 Interception 驱动方案，但不自动安装驱动。
- 生成统一运行报告、时间线、截图证据和脱敏日志。

验收标准：

- GUI 不再把 Maa Job 成功直接显示为业务操作成功。
- 失败报告能够回答失败步骤、使用的输入方式、验证证据和恢复方式。

### M7：发布与维护

目标：交付可安装、可升级的 Windows 桌面应用。

- 使用 PyInstaller 打包 GUI 和 MaaFramework 原生依赖。
- 将用户工作流、模板、设置和日志迁移到独立用户数据目录。
- 增加配置和工作流版本迁移。
- 增加 CI 测试、静态检查和打包验证。
- 提供示例工作流、首次启动引导和故障诊断入口。
- 评估自动更新或明确的手动升级流程。
- 发布前执行 Windows DPI、多显示器、中文路径和权限兼容性测试。

验收标准：

- 在未安装 Python 的 Windows 环境中可以启动。
- 升级不会覆盖用户工作流和模板。
- 安装包内 MaaFramework DLL、Qt 插件和字体资源完整。
